What this story does:

  2. Location field auto-fills when the user picks a workspace.

------------------------------------------------------------------------
BACKGROUND: WHAT IS A CATALOG CLIENT SCRIPT?
------------------------------------------------------------------------
A Catalog Client Script is JavaScript that runs in the user's browser
the instant a field changes — before anything is saved to the server.
It makes the form respond in real time.

Three script types:
  onChange  — Runs when a specific variable's value changes.
  onLoad    — Runs when the form first opens.
  onSubmit  — Runs when the user clicks Submit.

The g_form object is available automatically in all client scripts:
  g_form.getValue('variable_name')             — reads a field's current value
  g_form.clearValue('variable_name')           — empties a field
  g_form.showFieldMsg('name','msg','error')     — shows a red error below a field
  g_form.clearMessages()                        — removes all error messages
  g_form.setValue('variable_name', value)       — sets a field value

STEP 3.2: DATE VALIDATION CLIENT SCRIPT
------------------------------------------------------------------------
There is no purely click-and-configure way to compare two date fields
in real time on a catalog form in ServiceNow Zurich. A client script
is the right tool. Two options differ only in WHEN the validation fires.

  OPTION A: onChange — Validates the Moment the User Changes End Date
  (RECOMMENDED — real-time feedback, best user experience)
  -------------------------------------------------------------------
  Navigate: [All] > [Service Catalog] > [Catalog Client Scripts] > [New]
  OR type: `catalog_script_client.list` — Click [New].

  Field          Value
  -------------- -------------------------------------------------------
  Name           Validate Reservation Dates
  Catalog item   Workspace Reservation Request
  UI Type        All
  Type           onChange
  Variable name  end_date_time
  Active         Checked
  Application    Workspace Reservation Management

  Script:

    function onChange(control, oldValue, newValue, isLoading) {

    // Exit during form load or if the field was manually cleared by the user.
    if (isLoading || newValue === '') {
        return;
    }

    // Clear messages at the start of a fresh user interaction
    g_form.clearMessages();

    // Fetch the value of the start date field to compare against
    var start = g_form.getValue('start_date_time');

    // Nothing to compare if Start has not been set yet.
    if (start === '' || start === null) {
        return;
    }

    // ServiceNow dates are YYYY-MM-DD HH:MM:SS — sortable strings.
    // newValue < start means End is chronologically before Start.
    if (newValue < start) {
        
        // 1. Instantly clear the invalid value from the field.
        // This triggers the recursive loop immediately, which exits at the top check.
        g_form.clearValue('end_date_time');
        
        // 2. Wrap the message in a 0ms setTimeout.
        // This pushes the message rendering to the very end of the browser's execution queue,
        // ensuring it prints AFTER ServiceNow finishes clearing the field and wiping the UI.
        setTimeout(function() {
            g_form.showFieldMsg(
                'end_date_time',
                'End date/time should not be before the Start date/time.',
                'error'
            );
        }, 0); 
    }
}

  Click [Submit].

  WHY onChange IS RECOMMENDED:
  The error fires the moment the user selects a bad end time — they see
  the problem instantly and fix it before clicking Submit.

  -----------------------------------------------------------------------
  OPTION B: onSubmit — Validates Only When the User Clicks Submit
  (Simpler — use if you want to minimize client scripts)
  -----------------------------------------------------------------------
  Same navigation as Option A. Change these fields:

  Field          Value
  -------------- -------------------------------------------------------
  Type           onSubmit
  Variable name  (leave blank — onSubmit fires globally, not per variable)

  Script:

    function onSubmit() {
        var start = g_form.getValue('start_date_time');
        var end   = g_form.getValue('end_date_time');
        if (start && end && end < start) {
            g_form.addErrorMessage(
                'End date/time should not be before the Start date/time.'
            );
            return false;  // Blocks form submission
        }
    }

  LIMITATION: User fills in the full form, clicks Submit, then sees the
  error. Option A catches it in real time before Submit is ever clicked.

STEP 3.3: LOCATION AUTO-POPULATE
------------------------------------------------------------------------
When a user picks a workspace on the catalog form, the Location field
should auto-fill from that workspace's Location field.

  OPTION A: Configuration Method — No Script Record Created (RECOMMENDED)
  -------------------------------------------------------------------
  Uses the native Default value capability on the Location variable.
  The javascript: prefix tells ServiceNow to evaluate the expression
  at runtime. No separate client script record is created.

  1. Type `sc_cat_item.list` into the Filter Navigator.
     Open the Workspace Reservation Request catalog item.
  2. Click the [Variables] tab. Open the Location variable record.
  3. Set these two fields:
       Default value : javascript:current.variables.select_workspace.location
       Read only     : Checked
  4. Click [Update].

  What this expression does:
    current.variables.select_workspace.location
    reads the Location field from the workspace the user selected.

  KNOWN LIMITATION: The expression evaluates when the form first loads.
  If the user changes their workspace selection after the form opens,
  Location does NOT auto-update. For users who pick once and submit,
  this is acceptable. If you need dynamic updating, use Option B.

  -----------------------------------------------------------------------
  OPTION B: onChange Client Script — Dynamic, Updates on Every Change
  -----------------------------------------------------------------------
  Navigate: `catalog_script_client.list` — Click [New].

  Field          Value
  -------------- -------------------------------------------------------
  Name           Auto-Populate Location from Workspace
  Catalog item   Workspace Reservation Request
  UI Type        All
  Type           onChange
  Variable name  select_workspace
  Active         Checked
  Application    Workspace Reservation Management

  Script:

    function onChange(control, oldValue, newValue, isLoading) {
        if (isLoading || newValue === '') {
            g_form.clearValue('location');
            return;
        }
        g_form.getReference('select_workspace', function(workspace) {
            if (workspace && workspace.location) {
                g_form.setValue('location',
                    workspace.location,
                    workspace.dv_location || workspace.location
                );
            } else {
                g_form.clearValue('location');
            }
        });
    }

  If you implement Option B, remove the Default value expression from
  the Location variable to avoid the two methods conflicting.

Verify: Open catalog item in Service Portal as wrm_user.
        Select a workspace → PASS: Location field auto-fills.
        Set End before Start → PASS: Error fires, End field clears.
        Set valid End, Submit → PASS: RITM created.

PART D: BUILD THE discordNotification SCRIPT INCLUDE
------------------------------------------------------------------------
What is a Script Include?
  A reusable block of server-side JavaScript stored as a named record.
  Any script, flow, or business rule can call it by name. Think of it
  as a vending machine — build it once, and anything can deliver a
  Discord message without rewriting the same logic.

Navigate: [All] > [System Definition] > [Script Includes] > [New]
OR: `sys_script_include.list` — Click [New].

  Name             : discordNotification
  Application      : Workspace Reservation Management
  Accessible from  : All application scopes
  Active           : Checked
  Description      : Sends messages to Discord via webhook

Replace the default script with:

var discordNotification = Class.create();

discordNotification.prototype = {

    initialize: function() {
        // REPLACE x_#####_wrm.discord.webhook_url with YOUR full property name.
        this.webhookURL = gs.getProperty('x_#####_wrm.discord.webhook_url');
    },

    // message : text to post in the Discord channel
    // emoji   : optional emoji character prepended to the message
    sendMessage: function(message, emoji) {

        if (!this.webhookURL) {
            gs.warn('discordNotification: Webhook URL not configured. ' +
                    'Check the discord.webhook_url system property.');
            return false;
        }

        try {
            // sn_ws.RESTMessageV2 is ServiceNow's built-in HTTP client.
            var request = new sn_ws.RESTMessageV2();
            request.setEndpoint(this.webhookURL);
            request.setHttpMethod('POST');
            request.setRequestHeader('Content-Type', 'application/json');
            var payload = { content: (emoji ? emoji + ' ' : '') + message };
            request.setRequestBody(JSON.stringify(payload));
            var response   = request.execute();
            var httpStatus = response.getStatusCode();
            if (httpStatus == 200 || httpStatus == 204) {
                gs.info('discordNotification: Message sent successfully.');
                return true;
            } else {
                gs.error('discordNotification: Failed. HTTP status: ' + httpStatus);
                return false;
            }
        } catch (ex) {
            // Log and return false — do NOT crash the calling flow.
            gs.error('discordNotification: Exception: ' + ex.message);
            return false;
        }
    },

    type: 'discordNotification'
};

Click [Submit].

[Senior Note]: The try/catch block is critical. Without it, a failed
REST call throws an unhandled exception that halts the entire flow.
With try/catch, the error is logged and the flow continues normally.
Always wrap outbound HTTP calls in try/catch.

    (function execute(inputs, outputs) {
        try {
            var reservedFor = inputs.reserved_for
                ? inputs.reserved_for.getValue('name') : 'Unknown';
            var workspace = inputs.workspace
                ? inputs.workspace.getValue('workspace_name') : 'Unknown';

            // Convert raw date to human-readable display format.

                    '?sys_id=' + sysId;

                    '\n👋 Who can claim this one?';

                reservedFor + ' - ' + workspace + ' on ' + startDate +
                trackerLink;

            new discordNotification().sendMessage(message, '✅');

5. PHASE 3: ROLE-BASED ACCESS CONTROLS (STORY 17)
================================================================================
Update Set: WRM STRY0017 ACL Updates

What this story does:
  Reviews and corrects security rules on all three WRM tables so each
  role has exactly the access it needs — no more, no less.

------------------------------------------------------------------------
BACKGROUND: WHAT IS AN ACL?
------------------------------------------------------------------------
An ACL (Access Control List) is a security rule controlling whether a
user can create, read, write, or delete a record. Three checks in order:
  1. Role check      — Does the user have a required role?
  2. Condition check — Does the record match the filter?
  3. Script check    — If a script is on the ACL, does it return true?
All three must pass for access to be granted.

------------------------------------------------------------------------
REQUIRED ACL MATRIX
------------------------------------------------------------------------
WORKSPACE OPTIONS [x_#####_wrm_workspace_options]
  Read    No role required (anyone — needed for catalog reference picker)
  Create  wrm_admin or table role
  Write   wrm_admin or table role
  Delete  admin only

RESERVATION TRACKER [x_#####_wrm_reservation_tracker]
  Read    wrm_agent, wrm_user
  Create  wrm_agent, wrm_user
  Write   wrm_agent, wrm_user
  Delete  admin only

What this story does:

------------------------------------------------------------------------
BACKGROUND: WHAT IS AN SLA IN SERVICENOW?
------------------------------------------------------------------------
An SLA (Service Level Agreement) Definition is a configuration record
that creates a timer — called a Task SLA — on any record matching its
conditions. Think of it as a countdown clock.

  Task SLA  — the individual timer record attached to each matching RITM
  Target    — total allowed time (4 hours on a Business Hours schedule)
  Breach    — timer hits 100% without the stop condition being met
  Workflow  — background process that fires warning emails at 50/75/100%

There is no scripting alternative for SLA Definitions — this is a pure
configuration story.

STEP 9.8 (ARTIFACT D): SCRIPT INCLUDE — importOnDemand (GLOBAL SCOPE)
------------------------------------------------------------------------
WHY GLOBAL SCOPE?
  GlideImportSetLoader and GlideImportSetTransformer are global APIs.
  Your WRM app is scoped. Scoped apps can call global Script Includes
  but the Script Include itself must live in global scope to access them.
  This is the ONLY artifact in Story 21 that goes in global scope.

Navigate: `sys_script_include.list` — Click [New].

  Name         : importOnDemand
  API Name     : importOnDemand
  Application  : Global   ← CRITICAL: must be Global, not WRM
  Accessible   : All application scopes
  Active       : Checked

Script:

var importOnDemand = Class.create();

importOnDemand.prototype = {

    importAndTransform: function(dataSourceSysId, transformMapSysId) {

        // Step 1: Get the Data Source record
        var grDataSource = new GlideRecord('sys_data_source');
        if (!grDataSource.get(dataSourceSysId)) {
            gs.error('importOnDemand: Data Source not found: ' + dataSourceSysId);
            return null;
        }

        // Step 2: Create the Import Set and load the Excel attachment.
        // GlideImportSetLoader reads the file from the attachment
        // linked to the Data Source (File Retrieval Method = Attachment).
        var importSetLoader = new GlideImportSetLoader();
        var grImportSet = importSetLoader.getImportSetGr(grDataSource);
        importSetLoader.loadImportSetTable(grImportSet, grDataSource);

        // Step 3: Mark the import set as loaded before transforming.
        grImportSet.setValue('state', 'loaded');
        grImportSet.update();

        // Step 4: Run the Transform Map to create/update target records.
        var importSetTransformer = new GlideImportSetTransformer();
        importSetTransformer.setImportSetID(grImportSet.getUniqueValue());
        importSetTransformer.setMapID(transformMapSysId);
        importSetTransformer.setSyncImport(true); // wait for completion
        importSetTransformer.transformAllMaps(grImportSet);

        return { import_set_sys_id: grImportSet.getUniqueValue() };
    },

    type: 'importOnDemand'
};

Click [Submit].

  [API explained]:
    GlideImportSetLoader    — creates the import set and loads Excel rows
    getImportSetGr()        — creates a new import set GlideRecord
    loadImportSetTable()    — reads the file and populates staging table
    GlideImportSetTransformer — runs the Transform Map against staged rows
    setSyncImport(true)     — wait until transform is completely done

FLOW STEP 1 — Lookup Record: Find the Data Source
  Click [+] > [Action] > [Look Up Record]

  WHY: Avoids hard-coding a sys_id that breaks on migration.

FLOW STEP 2 — Lookup Record: Find the Attachment
  Click [+] > [Action] > [Look Up Record]
    Table      : Attachment [sys_attachment]
    Conditions:
      Table name   is   sc_req_item
      Table sys ID is   [Trigger > Requested Item Record > Sys ID]
    Store as   : Attachment Record

FLOW STEP 3 — Move Attachments: Move File to the Data Source
  Click [+] > [Action] > [Move Attachments]
    Source Attachment Record : [Step 2 > Attachment Record]
    Target record            : [Step 1 > Data Source Record]

  CRITICAL — Source field must be Step 2 > Attachment Record.
  Do NOT use Trigger > Requested Item Record for the source — that passes
  the RITM record (sc_req_item) but the action expects a sys_attachment
  record. Passing the RITM causes the error:
    "The Record type sc_req_item does not match with Attachment table type."

  WHY THIS STEP IS REQUIRED:
    The file was uploaded to the RITM (sc_req_item). GlideImportSetLoader
    reads from the DATA SOURCE record (File Retrieval Method = Attachment).
    Without this move, the loader finds nothing and silently imports zero
    rows with no error — a very hard failure to diagnose.
    After this step the file lives on the Data Source and the loader
    finds it correctly.

  [Senior Note]: Identified in the Kalisha_m Community article (flow images)
    and confirmed in the article comments by HulonB.

FLOW STEP 4 — Action: Import Workspace Data
  Click [+] > [Action] > [Import Workspace Data]
    Data Source Sys ID   : [Step 1 > Data Source Record > Sys ID]
    Transform Map Sys ID : (paste your Transform Map sys_id — static string)

  HOW TO GET TRANSFORM MAP SYS_ID:
    1. Type `sys_transform_map.list`.
    2. Open "WRM Workspace Options Bulk Transform."
    3. Copy sys_id from the URL bar.

FLOW STEP 5 — Lookup Record: Find Attachment on Data Source
  Click [+] > [Action] > [Look Up Record]
    Table      : Attachment [sys_attachment]
    Conditions:
      Table name   is   sys_data_source  (typed as literal text)
      Table sys ID is   [Step 1 > Data Source Record > Sys ID]
    Store as   : Data Source Attachment Record

  WHY: After the Move in Step 3, the file now lives on the Data Source
  record. We look it up here so we can delete it in Step 6.
  Using the Data Source as the parent (not the RITM) ensures we always
  find the right file even if a previous run left an old file behind.

FLOW STEP 6 — Delete Record: Clean Up the Attachment
  Click [+] > [Action] > [Delete Record]
    Record: [Step 5 > Data Source Attachment Record]

  WHY: Data hygiene — the file has been processed. Deleting from the
  Data Source record (not the RITM) ensures the Data Source is clean
  for the next run. If you delete from the RITM reference instead, old
  files can accumulate on the Data Source and get re-processed on the
  next import, causing duplicate or stale data.

FLOW STEP 7 — Lookup Record: Get Import Results
  Click [+] > [Action] > [Look Up Record]
    Table      : Transform History [sys_import_set_run]
    Conditions:
      Import set  is  [Step 4 > Action > Import Set Sys ID]
      State       is (one of)  complete, complete_error
    Store as  : Transform History Record

  WHY: sys_import_set_run holds row counts (Insert, Update, Error, Ignore).
  Used in the results email to give the manager actual numbers.

FLOW STEP 8 — Update Record: Close the RITM
  Click [+] > [Action] > [Update Record]
    Record : Trigger > Requested Item Record
    Field  : State
    Value  : Closed Complete

    Target record : [Trigger > Requested Item Record]

    Subject       : Workspace Options Bulk Import Complete —

  NOTE on Table and Target record: Always fill these in. They attach the

  Import Set has 0 rows / nothing imported, no error shown —

      Source Attachment Record = [Step 2 > Attachment Record]  (NOT the RITM)
      Target record = [Step 1 > Data Source Record]

      "The Record type sc_req_item does not match with Attachment table type."

  Workspace records duplicated on re-import —
    Coalesce not ticked on workspace_name field map. Open the Transform Map
    field maps, tick Coalesce on workspace_name, click OK on the index
    warning (this creates the database index — correct behavior).

    Old Excel file was not deleted from the Data Source record. The stale file
    is being re-processed. Verify Step 5 looks up sys_attachment where
    table_name = sys_data_source AND table_sys_id = [Step 1 > DS Sys ID],
    and Step 6 deletes [Step 5 > Data Source Attachment Record].

    inserted via the pill picker. Re-open Step 9, delete the typed tokens,
